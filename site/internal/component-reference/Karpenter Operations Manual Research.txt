Operations Manual: High-Performance Compute-as-a-Service with Karpenter 1.2.0
1. Executive Summary: The High-Performance Way
This document serves as the definitive "Day 2" Operations Manual for the "Compute-as-a-Service" research platform. It is architected to meet the rigorous demands of High-Performance Computing (HPC) and AI/ML workloads orchestrated by SkyPilot on Amazon EKS, utilizing Karpenter v1.2.0 as the exclusive node provisioning engine.
In the domain of HPC, infrastructure is not merely a utility; it is a performance primitive. The "High-Performance Way" dictates that every layer of the stack—from the kernel scheduler to the cloud control plane—must be optimized for throughput, latency, and resource efficiency. We do not tolerate "noisy neighbor" interference, I/O wait times on scratch storage, or suboptimal network placement. This manual translates high-level architectural requirements—specifically NVMe RAID0 ephemeral storage, Elastic Fabric Adapter (EFA) networking, and Cluster Placement Groups—into concrete operational workflows compliant with the Karpenter v1 API standards.1
Unlike legacy autoscalers that rely on the abstraction of Auto Scaling Groups (ASGs), Karpenter interacts directly with the EC2 Fleet API. This affords us sub-minute provisioning latency and precise "bin-packing" of heterogeneous instance types.2 However, with great power comes the requirement for precise configuration. The transition to Karpenter v1.0+ introduced significant API shifts, such as the removal of custom Launch Template support in favor of the strictly typed EC2NodeClass.3 This manual addresses these changes, providing the certified configurations necessary to maintain stability and performance in a production research environment.

2. Architectural Principles & System Design
2.1 The Compute-as-a-Service Tiered Architecture
The platform is designed as a tiered system where SkyPilot acts as the "Brain" (Orchestrator) and Karpenter acts as the "Muscle" (Provisioner).


Layer
	Component
	Role & Responsibility
	"High-Performance" Configuration
	Orchestration
	SkyPilot
	Job scheduling, multi-cloud abstraction, user interface.
	Submits Pods with explicit resource constraints (nvidia.com/gpu, vpc.amazonaws.com/efa).4
	Scheduling
	Kube-Scheduler
	Pod placement on existing nodes.
	Handles topology spread constraints and gang scheduling (if configured with Volcano/Coscheduler).
	Provisioning

	2.2 The Role of Karpenter 1.2.0
Karpenter v1.2.0 represents the first stable release line of the project. For our operations, this stability implies a lock on the API surface area for NodePool and EC2NodeClass resources. Crucially, v1.2.0 introduces support for Static Capacity (essential for handling reserved instances or specific capacity blocks) and Pod Level Resources, enhancing the granularity of bin-packing.1
Karpenter operates on a reconciliation loop:
1. Observe: Watch for pods marked as Unschedulable by the kube-scheduler.
2. Evaluate: specific constraints (Node Selectors, Affinities, Tolerations) and Resource Requests (CPU, Mem, GPU, EFA).
3. Provision: Calculate the optimal set of instances (Batching) that satisfy requirements at the lowest price (using Price-Capacity-Optimized strategies for Spot).6
4. Manage: Create NodeClaims which trigger EC2 RunInstances calls.
5. Consolidate: Actively monitor for underutilized nodes to disrupt and replace them with cheaper alternatives (Drift and Consolidation).2
2.3 The "Day 2" Operational Mindset
Day 2 operations focus on stability, observability, and optimization. In an HPC context, this means:
* Zero-Drift Configuration: Ensuring that the running state of the cluster matches the declarative configuration in Git.
* Interruption Resilience: Handling EC2 Spot Rebalance recommendations without crashing long-running training jobs.
* Startup Latency Minimization: Reducing the time from sky launch to GPU active by leveraging SOCI snapshotters and pre-provisioned AMI mappings.7

3. High-Performance Storage: NVMe RAID0 Strategy
Distributed training and genomic sequencing workloads often require massive high-IOPS scratch space. Relying on EBS (Network Attached Storage) for temporary data caching introduces latency and cost. The platform standard requires utilizing the local NVMe Instance Store available on d suffix instances (e.g., c5d.24xlarge, p4d.24xlarge) configured in RAID0 for maximum throughput.
3.1 The Challenge of Ephemeral Storage
By default, EC2 instances with multiple NVMe drives present them as separate block devices (/dev/nvme1n1, /dev/nvme2n1, etc.). The Kubernetes Kubelet, by default, utilizes the root volume (EBS) for /var/lib/kubelet and /var/lib/containerd. This results in:
1. I/O Contention: Image pulls and container logs compete with the OS.
2. Wasted Performance: The high-speed local NVMe drives remain unmounted and unused.
3. Capacity Limits: EBS root volumes are typically small (50-100GB), whereas local NVMe can provide Terabytes of space.
3.2 Karpenter v1.2.0 Implementation
Karpenter v1 simplifies this significantly compared to beta versions, but the implementation details vary strictly by the Operating System chosen: Bottlerocket or Amazon Linux 2023 (AL2023).
3.2.1 Configuration A: Bottlerocket (Preferred for AI/ML)
Bottlerocket is an immutable, API-driven OS designed for hosting containers. It is the preferred choice for our GPU nodes due to its minimized attack surface and seamless integration with EKS.
In Karpenter v1.2.0, the EC2NodeClass resource exposes a native instanceStorePolicy field. When set to RAID0, Karpenter automatically injects the necessary user data to stripe all available NVMe disks and mount them as the data partition for the container runtime.8
Manifest: Bottlerocket GPU NodeClass


YAML




apiVersion: karpenter.k8s.aws/v1
kind: EC2NodeClass
metadata:
 name: gpu-bottlerocket-high-perf
spec:
 amiFamily: Bottlerocket
 role: "KarpenterNodeRole-${CLUSTER_NAME}"
 subnetSelectorTerms:
   - tags:
       karpenter.sh/discovery: "${CLUSTER_NAME}"
 securityGroupSelectorTerms:
   - tags:
       karpenter.sh/discovery: "${CLUSTER_NAME}"
 
 # CRITICAL: Native RAID0 Support
 instanceStorePolicy: RAID0
 
 blockDeviceMappings:
   - deviceName: /dev/xvda
     ebs:
       volumeSize: 4Gi # Bottlerocket Root is small
       volumeType: gp3
       encrypted: true
   - deviceName: /dev/xvdb
     ebs:
       volumeSize: 20Gi # Data volume (metadata)
       volumeType: gp3
       encrypted: true
 
 # Fine-tuning kernel parameters for HPC networking
 userData: |
   [settings.kubernetes]
   "allowed-unsafe-sysctls" = ["net.ipv4.tcp_*", "net.core.somaxconn"]

Operational Insight: This configuration is only valid for Bottlerocket versions v1.22.0+. Using an older AMI with this policy will cause node join failures.8
3.2.2 Configuration B: Amazon Linux 2023 (AL2023)
AL2023 utilizes nodeadm, a YAML-based initialization system, replacing the legacy bash bootstrap scripts. While Karpenter v1 supports instanceStorePolicy: RAID0 for AL2023, complex HPC setups often require injecting additional custom configuration (e.g., installing specific monitoring agents or massive sysctl tuning) via userData.
The MIME Multipart Requirement:
When you supply custom userData and need Karpenter's native behavior (like cluster registration), you must structure the data as a MIME multipart document. You must also explicitly define the NodeConfig structure because providing custom user data disables Karpenter's automatic NodeConfig generation to prevent merge conflicts.9
Manifest: AL2023 HPC NodeClass with RAID0


YAML




apiVersion: karpenter.k8s.aws/v1
kind: EC2NodeClass
metadata:
 name: hpc-al2023-raid0
spec:
 amiFamily: AL2023
 role: "KarpenterNodeRole-${CLUSTER_NAME}"
 subnetSelectorTerms:
   - tags:
       karpenter.sh/discovery: "${CLUSTER_NAME}"
 securityGroupSelectorTerms:
   - tags:
       karpenter.sh/discovery: "${CLUSTER_NAME}"
 
 # Explicitly defined UserData requires manual NodeConfig
 userData: |
   MIME-Version: 1.0
   Content-Type: multipart/mixed; boundary="//"

   --//
   Content-Type: application/node.eks.aws
   apiVersion: node.eks.aws/v1alpha1
   kind: NodeConfig
   spec:
     cluster:
       name: ${CLUSTER_NAME}
       apiServerEndpoint: ${CLUSTER_ENDPOINT}
       certificateAuthority: ${CLUSTER_CA}
       cidr: ${SERVICE_CIDR}
     instance:
       localStorage:
         strategy: RAID0
     kubelet:
       config:
         maxPods: 110
   
   --//
   Content-Type: text/x-shellscript; charset="us-ascii"
   #!/bin/bash
   echo "Initializing High-Performance Tuning..."
   sysctl -w net.core.somaxconn=65535
   --//--

Troubleshooting: If nodes fail to join, check the karpenter_nodeclaims_registered metric. A persistent count of 0 created vs registered nodes usually implies a malformed NodeConfig (often missing apiServerEndpoint or certificateAuthority) when using custom userData.9
3.3 Validating Storage Performance
Once nodes are provisioned, SREs must validate the RAID0 configuration.
Runbook: Verification
1. Launch a debug pod on the node: kubectl debug node/<node-name> -it --image=amazonlinux:2023.
2. Inspect block devices: lsblk. You should see md0 (Software RAID) mounted at /mnt/k8s-disks/0 or similar paths utilized by Containerd.9
3. Check Kubelet root: df -h /var/lib/kubelet. It should report the size of the NVMe array (e.g., ~1.6TB for c5d.9xlarge), not the EBS root size.

4. High-Performance Networking: EFA & Placement Groups
For distributed training (e.g., PyTorch DDP, DeepSpeed), network latency is the bottleneck. The "High-Performance Way" mandates the use of Elastic Fabric Adapter (EFA) and Cluster Placement Groups.
4.1 Elastic Fabric Adapter (EFA) Integration
EFA bypasses the OS kernel networking stack using Libfabric to provide low-latency, high-bandwidth communication directly between applications.
4.1.1 Security Group Architecture (The Self-Reference Pattern)
EFA requires a security group that allows all traffic to and from itself. This is a non-negotiable requirement of the EFA protocol (SRD - Scalable Reliable Datagram).
* Action: Create an AWS Security Group (e.g., sg-efa-cluster).
* Inbound Rule: All Traffic, Protocol: All, Source: sg-efa-cluster (Self-reference).
* Outbound Rule: All Traffic, Protocol: All, Destination: sg-efa-cluster (Self-reference).
4.1.2 Karpenter EFA Configuration
Karpenter v1 does not create security groups; it consumes them. The EC2NodeClass must use securityGroupSelectorTerms to attach both the Cluster SG (for Kubelet/Control Plane communication) and the EFA SG.
Operational Best Practice:
Tag your Security Groups explicitly for Karpenter discovery.
* Cluster SG Tag: karpenter.sh/discovery: ${CLUSTER_NAME}
* EFA SG Tag: karpenter.sh/discovery: ${CLUSTER_NAME}, purpose: efa


YAML




spec:
 securityGroupSelectorTerms:
   # This selects ALL SGs with the discovery tag.
   # Ensure ONLY appropriate SGs have this tag to avoid attaching unrelated rules.
   - tags:
       karpenter.sh/discovery: "${CLUSTER_NAME}"

Insight: Karpenter performs an logical AND on tags within a list item. If you have separate tags, ensure your selector logic is robust. Using a single discovery tag on all required groups is the simplest and most robust method.9
4.1.3 Pod Scheduling for EFA
SkyPilot automatically injects the resource limit vpc.amazonaws.com/efa into the Pod spec when EFA is requested in the task YAML. Karpenter observes this requirement and filters the instance types to only those supporting EFA (e.g., p4d, trn1, c6gn).
Verification: kubectl describe pod <pod-name> should show:


YAML




Requests:
 vpc.amazonaws.com/efa: 1 # or more

If Karpenter fails to provision, verify that the NodePool requirements include EFA-capable instance families.12
4.2 Cluster Placement Groups
To minimize physical network distance between nodes, we utilize AWS Placement Groups with the cluster strategy. This packs instances close together inside an Availability Zone.
4.2.1 NodePool Configuration
In Karpenter v1.2.0, placement group support is defined in the NodePool spec (specifically under spec.placementGroup) or implicitly handled if the provider supports it, but the definitive method for EKS Auto Mode or advanced Karpenter configs is increasingly moving to explicit definitions. Note that research snippets indicate placementGroup configuration support via the NodePool spec.13
Manifest: NodePool with Placement Group


YAML




apiVersion: karpenter.sh/v1
kind: NodePool
metadata:
 name: gpu-cluster-placement
spec:
 template:
   metadata:
     labels:
       topology.kubernetes.io/zone: us-east-1a # Placement groups are AZ-specific
   spec:
     nodeClassRef:
       group: karpenter.k8s.aws
       kind: EC2NodeClass
       name: hpc-bottlerocket
     requirements:
       - key: karpenter.k8s.aws/instance-generation
         operator: Gt
         values: ["4"]
     # Explicit Placement Group definition
     placementGroup:
       groupName: my-hpc-cluster-pg
       strategy: cluster

Operational Warning: cluster strategy placement groups are notoriously difficult to satisfy with Spot capacity because they require large contiguous blocks of hardware. InsufficientInstanceCapacity errors are highly likely.
Mitigation: Use this only for capacity-type: on-demand or Reserved Instances (ODCR). For Spot, rely on AZ-level affinity without strict placement groups to maximize fulfillability.15

5. Image Acceleration: SOCI and Parallel Pulls
In HPC, "Time to Science" includes the node startup time. Pulling massive Docker images (e.g., 20GB+ NVIDIA CUDA images) can take minutes. Seekable OCI (SOCI) is an AWS technology that enables lazy loading of container images, allowing the application to start before the full image is downloaded.
5.1 SOCI Architecture
SOCI creates an index (zTOC) of the image layers, allowing containerd to extract individual files on demand. For Karpenter nodes, we configure the container runtime to utilize the soci-snapshotter.
5.2 Bottlerocket Configuration for SOCI
Bottlerocket v1.19.0+ includes the SOCI snapshotter. We must enable it via the EC2NodeClass settings.7
Manifest Update: Enable SOCI


YAML




apiVersion: karpenter.k8s.aws/v1
kind: EC2NodeClass
spec:
 #... other config...
 userData: |
   [settings.container-runtime]
   snapshotter = "soci"
    
   [settings.container-runtime-plugins.soci-snapshotter]
   pull-mode = "parallel-pull-unpack"

Insight: The parallel-pull-unpack mode is crucial for HPC. Unlike pure lazy loading (which can cause runtime latency spikes when a file is accessed), parallel pull aggressively downloads chunks in the background while prioritizing the files needed now. This hybrid approach balances startup speed with runtime consistency.7

6. Compute Capacity Management: Spot, Reservations, and Consolidation
The platform's economic model relies on the intelligent orchestration of purchasing options.
6.1 Spot Instance Orchestration
Karpenter v1 uses the Price-Capacity-Optimized (PCO) allocation strategy for Spot instances. This strategy does not just pick the cheapest instance; it picks the instance from the pool with the deepest capacity that is also cost-effective. This dramatically reduces the rate of SpotInterruption events.6
NodePool Strategy:
Define broad requirements to give PCO the maximum search space.


YAML




requirements:
 - key: karpenter.k8s.aws/instance-family
   operator: In
   values: ["p4d", "p3dn", "g5"] # Allow fallback to older gens if P4 is full
 - key: karpenter.sh/capacity-type
   operator: In
   values: ["spot"] 

6.2 On-Demand Capacity Reservations (ODCR) & Static Capacity
For critical jobs that cannot tolerate interruption, we utilize Capacity Reservations (ODCR). Karpenter v1.2.0 introduces support for selecting these reservations. While fully native support (Capacity Blocks) matures in v1.3/v1.6, v1.2.0 allows selecting specific reservations via capacityReservationSelectorTerms in the EC2NodeClass.18
Manifest: Targeting a Capacity Reservation


YAML




apiVersion: karpenter.k8s.aws/v1
kind: EC2NodeClass
spec:
 capacityReservationSelectorTerms:
   - id: cr-0123456789abcdef0 # Explicit ID of the reservation
   # OR
   - tags:
       Workload: "Strategic-Research"

Operational Workflow: When a high-priority SkyPilot task is launched (tagged with priority: high), the scheduler targets a NodePool associated with this EC2NodeClass. Karpenter ensures the node is launched into the pre-paid, guaranteed capacity.19
6.3 Interruption Handling & SQS
Karpenter consumes Spot Interruption warnings from an SQS queue.
1. Event: AWS sends a "Rebalance Recommendation" or "Interruption Warning" (2 min notice).
2. Reaction: Karpenter taints the node karpenter.sh/disruption=disrupting:NoSchedule.
3. Action: It immediately cordons and drains the node.
4. Rescheduling: The Pod is evicted, becomes Pending, and triggers a new NodeClaim instantly.
Configuration: Ensure the Karpenter Helm chart is installed with settings.interruptionQueueName set to the SQS queue receiving these EventBridge rules.5

7. SkyPilot Integration Guide
SkyPilot is the user-facing interface. It abstracts the Kubernetes details, but understanding its interaction with Karpenter is vital for "Day 2" debugging.
7.1 Mapping Task Resources to NodePools
When a user runs sky launch, SkyPilot generates a Pod spec.
* SkyPilot Request: accelerators: V100:8
* Pod Spec:
YAML
resources:
 limits:
   nvidia.com/gpu: 8
nodeSelector:
 skypilot.co/accelerator: v100

* Karpenter Matching: Karpenter looks for a NodePool that can satisfy nvidia.com/gpu: 8. It creates a p3.16xlarge (which has 8 V100s).
7.2 Multi-Tenancy and Cost Allocation
To separate research teams, use SkyPilot's labels feature in the task definition to drive Karpenter's NodePool selection.
SkyPilot Task:


YAML




resources:
 labels:
   team: genomics-lab

Karpenter NodePool:


YAML




apiVersion: karpenter.sh/v1
kind: NodePool
metadata:
 name: genomics-pool
spec:
 template:
   metadata:
     labels:
       billing-team: genomics-lab # Propagated to EC2 tags for Cost Explorer
   spec:
     requirements:
       - key: team
         operator: In
         values: ["genomics-lab"]

This ensures that the "Genomics Lab" budget pays for these specific nodes.

8.2 Runbook: Handling "Insufficient Capacity" Errors
Symptom: Pods stuck in Pending. Karpenter logs show InsufficientInstanceCapacity errors from the Fleet API.22
Diagnosis: The specific Instance Type + AZ combination is sold out in AWS.
Resolution Steps:
   1. Check Diversity: Does the NodePool allow other instance types? (e.g., if p4d is requested, is p5 or p3 allowed?).
   2. Check Zones: Are we restricted to us-east-1a? Expand requirements to include other zones.
   3. Fallback: If the workload is critical, manually patch the NodePool to allow on-demand capacity type, or rely on a NodePool that allows both spot and on-demand (Karpenter will try Spot first, then fallback).
8.3 Runbook: Debugging Zombie Nodes
Symptom: Karpenter creates a NodeClaim, the EC2 instance boots, but never appears in kubectl get nodes.
Investigation:
   1. Check IAM: Does the KarpenterNodeRole have eks:DescribeCluster? (Required for nodeadm/bootstrap).
   2. Check Tags: Does the EC2 instance have the tag kubernetes.io/cluster/${CLUSTER_NAME}: owned?
   3. Check Network: Is the instance in a private subnet with a route to the NAT Gateway or VPC Endpoints for EKS, STS, and EC2?.5
   4. Check UserData: If using AL2023 with custom UserData, did you include the MIME multipart NodeConfig with the cluster details? (Common failure point).11
8.4 Upgrading Karpenter
Upgrading from v1.x to v1.y is generally safe but requires strict adherence to the compatibility matrix.23
   1. Backup: Export existing NodePools and EC2NodeClasses.
   2. Helm Upgrade: Update the CRDs first, then the Helm chart.
   * kubectl replace -f https://raw.githubusercontent.com/aws/karpenter-provider-aws/v${VERSION}/pkg/apis/crds/karpenter.sh_nodepools.yaml
   * helm upgrade karpenter...
   3. Drift Check: Karpenter v1 is sensitive to Drift. An upgrade might trigger a rolling update of all nodes if the new version changes the default User Data hash.
   * Control: Set karpenter.sh/do-not-disrupt: "true" on critical nodes before upgrading to prevent mass-recycling.

9. Conclusion
The "High-Performance Way" is not a static configuration; it is an operational discipline. By leveraging Karpenter v1.2.0's advanced capabilities—RAID0 Instance Store policies, EFA-aware scheduling, and Spot-to-Spot consolidation—we create a research platform that is both economically viable and performant.
The SRE's role shifts from managing servers to managing constraints. The NodePool is the contract between the researcher's need and the cloud's capacity. This manual provides the codified knowledge to enforce that contract, ensuring that when a researcher asks for "Compute," the platform delivers it—instantly, efficiently, and at scale.
